• Findings

  - High: support selection is not actually applied in the forward pass, so “active columns” are still all contributing activations. The example graph only
    connects pixels -> columns -> aggregator -> output and never feeds a support mask into columns.slot("mask") or aggregator.slot("mask")
    (split_mnist_continual.py:158, split_mnist_continual.py:169). In ColumnNode.forward(), missing mask input defaults to an all-ones mask, so every column
    stays live (nodes.py:297). That means support selection only affects gradient masking, not routing, and frozen/stale columns can still interfere with
    logits.
  - High: the causal support audit is generating synthetic labels instead of measuring real swapped-support behavior. After collecting baseline losses,
    _run_support_swap_audit() stops evaluating actual swap variants and replaces them with estimated_gain = (swap_in_score - swap_out_score) * 0.1 + noise
    derived from support-bank averages plus Gaussian noise (trainer.py:1177). Those fabricated gains are then fed into the causal predictor/trust pipeline, so
    the “causal” learner is training on noisy heuristics rather than observed counterfactual outcomes.
  - High: TransWeave state is also synthetic, so the transfer mechanism is decoupled from the learned model. _register_transweave_task_end() explicitly says it
    is generating representative composer state “for now,” then builds random projections with np.random.randn(...) and shell activities from hand-written
    heuristics/noise instead of model parameters or measured activations (trainer.py:1376, trainer.py:1397, trainer.py:1483). Any transfer stats or downstream
    task adaptation attributed to TransWeave are therefore not grounded in what the network actually learned.
  - Medium: optimizer state is effectively thrown away at every task boundary and on checkpoint restore. SequentialTrainer constructs self.optimizer, but
    train_task() calls train_backprop() / train_pcn() which each reinitialize opt_state = optimizer.init(params) internally (trainer.py:747,
    train_backprop.py:171, train.py:161). Checkpoints also save params but not optimizer state, and load_checkpoint() just re-inits it again (trainer.py:1661,
    trainer.py:1698). That makes resume non-faithful and wipes Adam moments across tasks, which changes continual-learning behavior.
